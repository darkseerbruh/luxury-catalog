## Git / workflow (locked)
- **`main` is the single source of truth.** Sync from `main` at session start, merge back to `main` at end (see AGENTS.md "Branch & sync workflow"). Never build off old per-session branches.
- She prefers **automation and guardrails** so she doesn't have to remember process.
- **Recurring manual ops should become one-button automations.** She pushes back on by-hand steps (*"why do I have to do this by hand?"*) — turn them into browser-triggered automations she can self-serve with no laptop/CLI. Concrete: **DB migrations now run via a GitHub Action she triggers** (`.github/workflows/db-migrate.yml`, with an optional `repair_versions` input to reconcile the recurring "objects exist but the migration history wasn't recorded" mismatch). Prefer building this over handing her CLI commands. **Never edit a migration that's already been applied** — Supabase `db push` skips applied versions, so edits silently don't land; add a *new* migration for the delta (e.g. 0022 was edited after it ran → the new columns had to move to 0023, with `IF NOT EXISTS` guards). Verify the live columns afterward rather than assuming the run took.
- **Merging to `main` ships live** (Vercel tracks `main`), so she treats merge-to-`main` as a **deploy gate she approves explicitly.** Present finished work on the per-session branch *with verification* (`tsc`/`eslint`/`next build`/tests green + git proof), then merge on her go-ahead — often a one-word *"Merge."* Don't silently auto-merge to `main`. She'll also spot-check that it actually landed (*"is this on main and pushed to GitHub?"*) and that a specific surface really changed (*"did you change the home page copy?"*) — answer with ground truth (git log / origin sync), not assertion.
- **Expect `main` to move under you between merges** — parallel sessions land PRs constantly. Re-sync and re-check for conflicts/drift right before merging; a clean merge of someone else's PR can still pull in new work that needs your pass (see voice gate below).
- **Don't let a merge break her live app before she's applied the migration.** She applies DB migrations herself (human-gated, often a while after the code merges), so **thread new DB-dependent features behind resilient reads** (catch the missing-column error and return empty → the feature shows a placeholder/“no data yet” instead of 404-ing). Migrations ship in the same PR with clear "apply this, then it activates" notes; the app degrades gracefully until she does.
- **Migration ordering gotcha (learned 2026-06-23):** the "Apply database migrations" GitHub Action applies whatever migration files are **on `main`**. A new migration must be **merged to `main` BEFORE** she runs the Action — running it while the migration is still on a branch silently applies nothing new, and a later prod write expecting the new index/column fails confusingly. Sequence: merge → run the Action → then the prod load that depends on it.
- **Parallel background agents to keep the chat free (she asked for this 2026-06-23):** for several independent build tasks, spawn **worktree-isolated background agents** (one per task, non-overlapping files) so she keeps using the main chat. Worktrees don't get `.env.local`, so agents can't touch prod — code+tests only; prod writes stay with you/her. Verify each branch's gates yourself before recommending merge; merge stays her deploy gate.

## Surfacing features on the home/entry pages (UX copy)
- **Lead with the value prop, not a list.** On entry surfaces (home, nav), a flat list of feature links/chips is *"not a good experience."* Each use case needs a **hook that sells why it's for you** and drives traffic to the feature — motivate the click, don't just enumerate.
- **One entry per destination — dedupe overlapping flows.** If two use cases point at the same place (e.g. two chips both linking to `/identify`), **show it once** as a single combined value-prop block with copy that speaks to *both* audiences. Never surface duplicate links to the same flow.
- **Offer a few copy/positioning options to choose from.** She'll ask you to *"suggest a few options for how to best highlight"* something — give 2–3 distinct angles per item, ship a sensible default, and record the alternatives where sessions can reuse/swap them (e.g. `docs/ux/home-use-case-value-props.md`). Treat copy as A/B-swappable, not final.
